CRA Article 14

Active exploitation notification, KEV correlation, and cryptographically chained audit evidence for the Cyber Resilience Act.

September 2026 deadline. CRA Article 14 enters force in September 2026. If your software ships to the EU market, you need a demonstrable notification pipeline for actively exploited vulnerabilities — not just a policy, but evidence of detection, assessment, and decision.

What the regulation requires

Article 14 of the EU Cyber Resilience Act obligates manufacturers of products with digital elements to:

  1. Monitor for actively exploited vulnerabilities affecting their products
  2. Notify ENISA within 72 hours of confirmed active exploitation (early warning)
  3. Submit a detailed report within the regulatory deadline
  4. Maintain an audit trail of all decisions and notifications

The key operational challenge is not detecting the CVE — it is proving, on paper, that you detected it, assessed it, and acted within the regulatory windows.

KEV correlation

Wardex correlates any scanner output against the CISA Known Exploited Vulnerabilities catalogue:

curl -sSL https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json \
    -o kev-catalogue.json

wardex convert grype grype-output.json --kev kev-catalogue.json

The output is a structured YAML file with each vulnerability annotated with actively_exploited, KEV dateAdded, and CISA notes — ready for gate evaluation.

Active exploitation hard stop

When a vulnerability is actively exploited, Wardex exits with code 12:

wardex evaluate --evidence vulns.yaml --config wardex-config.yaml

# Exit code: 12 (ActivelyExploited)

The 12 exit code is distinct from the normal gate block (10). It triggers a different CI/CD path because the regulatory clock has started:

  • Generates a CRA Article 14 notification artifact signed with HMAC-SHA256
  • Records a chained audit entry tracking the three CRA deadlines
  • Cannot be overridden by risk acceptance — the regulation does not permit it

Artifact lifecycle

# List all artifacts
wardex art14 list

# Show a specific artifact
wardex art14 show <artifact-id>

# Verify HMAC integrity
wardex art14 verify <artifact-id>

# Mark early-warning notification as dispatched
wardex art14 mark-dispatched <artifact-id> --phase early-warning

# Close the case with patch confirmation
wardex art14 finalize <artifact-id> --patch-date 2026-06-09T12:00:00Z

Each artifact is cryptographically chained to the previous one using HMAC-SHA256, producing an append-only audit trail. This is what a regulator asks for — not a Jira screenshot, but a chain of custody from detection to patch confirmation.
