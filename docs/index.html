Wardex

Risk-driven release gate and compliance gap analyser. Turns vulnerability scanners into CRA-compliant audit evidence.

GitHub Changelog CRA Article 14
Install Quickstart

What it does

  • Correlates scanner output against the CISA KEV catalogue to identify actively exploited CVEs
  • Generates signed CRA Article 14 notification artifacts with HMAC-SHA256 audit chain
  • Evaluates release risk per asset context — the same CVE can ALLOW, WARN, or BLOCK depending on criticality, exposure, and compensating controls
  • Cross-references documented vs implemented controls against ISO 27001, NIS2, DORA, and SOC 2 catalogues
  • Returns distinct exit codes for each outcome: gate block (10), compliance gap (11), active exploitation (12)

Supported frameworks

ISO/IEC 27001:2022

Annex A control mapping with maturity scoring and evidence classification.

NIS 2

Cyber hygiene, supply chain, and incident reporting controls for essential entities.

DORA

ICT risk management, digital operational resilience, and third-party oversight.

SOC 2

Trust services criteria mapped to documented and implemented control layers.

Quickstart

# Install
go install github.com/had-nu/wardex@v2.0.0

# Convert scanner output to Wardex format
wardex convert grype test/usability/grype-results.json > vulns.yaml

# Evaluate with asset context
wardex evaluate \
  --evidence vulns.yaml \
  --config doc/examples/wardex-config.yaml

Exit codes: 0 ALLOW · 10 BLOCK · 11 Compliance Gap · 12 Active Exploitation

Install

Requires Go ≥ 1.26.

go install github.com/had-nu/wardex@v2.0.0

Or build from source:

git clone https://github.com/had-nu/wardex.git
cd wardex && make build

CRA Article 14

CRA Article 14 notification obligations enter force September 2026. Wardex implements the full notification pipeline with KEV correlation, distinct exit code 12, and HMAC-SHA256 signed artifacts.
