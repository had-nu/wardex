Release Gate Risk Model

Compound risk calculation that determines BLOCK / WARN / ALLOW per vulnerability.

R(α) = (CVSS/10 × EPSS) × (1 − Σ(Φ)) × C(α) × E(α)
CVSS/10 Normalised base score [0,1] EPSS Exploit probability [0,1] (defaults to 1.0 if missing) Σ(Φ) Compensating control effectiveness [0,0.8] C(α) Asset criticality [0,1] E(α) Exposure = internetWeight × (1 − authRed) × (1 − reachableRed)

CRA Article 14 Notification Pipeline

EU Cyber Resilience Act: active exploitation detection with HMAC-signed artefacts and regulatory deadlines.

Active Exploitation Detected

CISA KEV matching or manual classification. Wardex generates a signed notification artefact.

Exit code 12 — hard stop

Early Warning (+24h)

Submit early warning notification to competent authority (CSIRT/ENISA).

Deadline: awareness timestamp + 24 hours

Notification (+72h)

Formal vulnerability notification with product details, impact assessment, and mitigation timeline.

Deadline: awareness timestamp + 72 hours

Mark Dispatched

wardex art14 mark-dispatched <id> — acknowledges notification sent to authority.

Finalize (+14d after patch)

wardex art14 finalize <id> --patch-date ... — close-out report with vulnerability description, severity, impact, threat actor details, and corrective measures.

Deadline: 14 days after corrective measures available

🔐 Artefact Signing & Verification

Every notification artefact is HMAC-SHA256 signed using WARDEX_ACCEPT_SECRET. The artefact contains CVE IDs, awareness timestamp, deadlines, and a chained audit trail. Verification:

wardex art14 verify <artefact-id> wardex art14 list → artefact metadata & HMAC status

🧩 Correlation Engine

Existing controls are matched to catalog controls via keyword overlap and domain intersection. Each match gets a confidence label (high or low). The scoring model uses min-confidence flag (default: low) to filter matches before gap analysis.

Trust System Sealed Configs

Ed25519 keypairs, revocable trust store, and signed configuration states for non-repudiation in CI/CD pipelines.

🔑 Key Generation

wardex keygen generates Ed25519 keypairs with human-readable role labels. Keys are stored in ~/.wardex/keys/ with restricted permissions (0600).

Ed25519Role labels0600 permissions

📜 Trust Store

Signed registry of trusted public keys with roles, issuance dates, and optional KERI-compatible references. Supports key rotation and targeted revocation.

KERI refsKey rotationRevocation

🔒 Config Seal

wardex config seal signs the active YAML config with a trusted key. The sealed payload (.wexstate) includes the original config, signer identity, key ID, and timestamp. wardex evaluate verifies the seal before loading.

.wexstateIntegrityFailure on broken sealCI enforcement

🛡️ Non-Repudiation in CI

When --strict is used, unsealed YAML configs are rejected with exit code 3. All gate decisions are logged to a chained audit log with evidence hash, config hash, and decision details.

Evidence hash (SHA256)Chained audit logExit code 3
