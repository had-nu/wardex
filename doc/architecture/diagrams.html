Análise Sumária — A Lacuna de Estado

O Que Falta

O Wardex foi concebido como um sistema stateless — cada execução é uma ilha. Esta arquitectura foi uma escolha deliberada para maximizar a portabilidade e minimizar dependências operacionais. No entanto, esta simplicidade cria uma lacuna significativa quando o sistema é integrado em pipelines de conformidade contínua:

CapacidadeActual (Stateless)Proposto (Com State Store)
Histórico de decisõesJSONL sem query — busca manualÍndice por data, CVE, decisão — query instantânea
Trend analysisImpossívelGráfico 30/60/90 dias — tendência de risco
Alertas proactivosNenhumAcceptances a expirar, risco crescente, gaps novos
Rollback de configImpossível — snapshots sobrescritosVersões anteriores preservadas, diff entre versões
DashboardNão suportadowardex state dashboard → HTML interactivo
Multi-execução aggregationCada execução é independentewardex state aggregate → relatório consolidado

Porque É Importante

Num contexto regulatório (NIS2, DORA, CRA), a capacidade de demonstrar melhoria contínua da postura de segurança é tão importante quanto a postura actual. Sem um state store, o Wardex pode dizer "esta é a sua conformidade hoje" mas não pode dizer "a sua conformidade melhorou 15% nos últimos 90 dias" — exactamente o tipo de evidência que os auditores e reguladores procuram.

Proposta

O componente pkg/statestore/ proposto é opt-in (sem breaking changes), file-based (sem base de dados externa), e usa apenas stdlib Go (zero novas dependências). A integração é mínima: uma chamada a RecordDecision() após cada gate.Evaluate(), e uma chamada a SaveState() no final de cada execução. O ganho é transformar o Wardex de uma ferramenta pontual num sistema de conformidade contínua com memória.
