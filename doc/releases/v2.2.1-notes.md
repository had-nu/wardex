# Wardex v2.2.1 Release Notes

**Release Date:** 2026-06-26  
**Version:** 2.2.1  
**Theme:** CRA-Ready Release Gate ‚Äî Explicit Logging & Persistent State

---

## Highlights

### Feature A: Explicit Rejection Logging

All 26 silent rejection behaviors now produce explicit stderr logging. No more hidden filtering ‚Äî every decision is visible in CI/CD output.

**Impact:** Security teams can now audit all filtering decisions without code inspection.

### Feature B: Persistent State Store

Cross-execution memory with BLAKE3 hash chain integrity and optional WORM protection. Track risk trends, view decision history, and verify audit trail integrity.

**Impact:** First step toward trend analysis, proactive alerts, and multi-execution dashboards.

---

## Detailed Changes

### Feature A: Explicit Rejection Logging

#### CRITICAL (3 locations)
- `accept.Load()` now accepts `io.Writer` parameter and logs all rejections
- `cmd/evaluate/evaluate.go` adds `[WARN]` when `ResolveSecret` fails
- `main.go` adds `[WARN]` when acceptances cannot be loaded

#### HIGH (10 locations)
- EPSS `FetchScores()` logs malformed scores and out-of-range values
- CycloneDX SBOM logs empty IDs and missing CVE scores
- Grype converter logs empty and duplicate CVE skips
- Art14 artefacts logs malformed file skips
- OpenVEX logs unrecognized state drops
- Policy loader logs new file creation

#### MEDIUM (11 locations)
- Trust store logs admin key decode failures
- Snapshot delta logs missing controls
- Maturity scorer logs unknown domains
- Analyzer gap logs nonexistent controls
- KEV converter logs missing entries
- CPL verifylink logs empty config hashes

#### New Files
- `pkg/ui/logging.go` ‚Äî Unified logging helpers (`LogReject`, `LogWarn`, `LogInfo`, `LogHint`)

#### Output Format
All messages use standardized prefixes:
- `[FAIL]` ‚Äî Fatal error preventing continuation
- `[WARN]` ‚Äî Operational warning (non-blocking)
- `[INFO]` ‚Äî Contextual information
- `[HINT]` ‚Äî Resolution suggestion

---

### Feature B: Persistent State Store

#### Core Statestore (`pkg/statestore/`)
- `state.go` ‚Äî Types: `State`, `TrendPoint`, `TrendAnalysis`, `TrendDirection`
- `store.go` ‚Äî Main API: `New`, `LoadState`, `SaveState`, `RecordDecision`, `History`, `TrendAnalysis`, `Cleanup`, `VerifyChain`
- `chain.go` ‚Äî BLAKE3 hash chain: `ChainEntry`, `ComputeChainHash`, `HashBytes`, `LoadChain`, `SaveChain`, `VerifyChain`, `AppendEntry`
- `worm.go` + platform files ‚Äî WORM protection (immutable bit on Linux/macOS, readonly on Windows)
- `history.go` ‚Äî Historical queries: `ListHistory`, `HistoryBetween`, `HistoryCount`
- `trend.go` ‚Äî Formatting: `FormatTrend`, `FormatHistory`, `FormatDashboard`

#### CLI Commands (`cmd/state/`)
- `wardex state status` ‚Äî Current state and chain integrity
- `wardex state history` ‚Äî Decision history
- `wardex state trend` ‚Äî Risk trend analysis
- `wardex state dashboard` ‚Äî Comprehensive dashboard
- `wardex state verify` ‚Äî BLAKE3 chain verification
- `wardex state cleanup` ‚Äî Remove old history snapshots

#### Configuration
```yaml
state_store:
  enabled: true
  dir: .wardex
  retention_days: 90
  worm: true
```

#### Integration
- `config/config.go` ‚Äî Added `StateStoreConfig` struct
- `cmd/evaluate/evaluate.go` ‚Äî Records decisions when `state_store.enabled=true`, `--trend` flag
- `main.go` ‚Äî Registered `wardex state` command

## Migration Guide

### From v2.2.0 to v2.2.1

**No breaking changes.** Both features are opt-in:

1. **Explicit Logging** ‚Äî Automatic. All existing behavior preserved, just with added logging.

2. **State Store** ‚Äî Opt-in via configuration:
   ```yaml
   state_store:
     enabled: true
   ```

### Upgrading

```bash
# Download v2.2.1
go install github.com/had-nu/wardex/v2@v2.2.1

# Verify installation
wardex --version
```

## Known Limitations

1. **WORM on macOS** ‚Äî Requires `FS_IOC_SETFLAGS` support (available on HFS+ and APFS)
2. **WORM on Windows** ‚Äî Uses `FILE_ATTRIBUTE_READONLY` (not true immutability)
3. **Trend analysis** ‚Äî Requires at least 2 data points for meaningful analysis
4. **Chain verification** ‚Äî Full chain must be present (no partial verification)
