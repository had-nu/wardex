## Wardex v1.9.1 — Schema Consistency Cleanup

Patch release that aligns `wardex-config.yaml` with what the code actually consumes.
No behavioural change. No migration required.

The audit that prompted this release found five configuration blocks declared in the
schema but never read by any active code path. Users who calibrated those fields
carefully observed no effect — a footgun for any organisation adopting Wardex,
regardless of sector or maturity. v1.9.1 removes the orphan fields and corrects the
documentation that mis-taught them.

### Removed

The following configuration blocks had no consumer in the active codebase. They have
been removed from `config/config.go` and from documentation/examples:

- **`organization`** (`name`, `sector`, `scope`) — informational only; never read.
- **`domain_weights`** — placeholder for an unshipped feature; removed pending a
  concrete use case.
- **`control_weights`** — placeholder; same rationale.
- **`thresholds`** (top-level: `fail_above`, `warn_above`) — duplicated
  `release_gate.warn_above` semantically and was never read. The live threshold field
  is `release_gate.warn_above`. The live failure-on-gap flag is `--fail-above` on
  `wardex evaluate` (CLI), not a YAML field.
- **`reporting.verbose`** — the CLI flag `--verbose` in `main.go` is the source of
  truth.

### Fixed

- **`doc/examples/wardex-config.yaml`** was teaching users to write a `thresholds:`
  block whose keys (`critical_vulnerabilities`, `high_vulnerabilities`) did not match
  any field in the codebase. The example now mirrors the live schema: `release_gate`,
  `acceptance`, `reporting`, and `profiles`.
- **`test/testdata/wardex-config.yaml`** had three blocks that misled anyone reading
  the repository to learn the schema: `organization:` and `domain_weights:` (orphan),
  and `accept:` (wrong key — the schema field is `acceptance:`, and the HMAC secret
  is read from the `WARDEX_ACCEPT_SECRET` env var, not from YAML). All three removed.

### Notes

- `internal/SPEC_wardex_trust_.md` contained an example `domain_weights.technological:
  "PENDING_APPROVAL"` for the `wardex config seal` command. That reference is now
  historical; the SPEC has been annotated to point readers at a live field
  (`release_gate.risk_appetite`) for equivalent examples.
- A regression test in `config/config_v191_test.go` ensures legacy YAMLs (with the
  now-removed blocks) continue to parse without error.

### Backward compatibility

`wardex-config.yaml` files written for v1.9.0 with any of the removed blocks continue
to load without error in v1.9.1. The Go YAML decoder ignores unknown fields by default,
so these blocks were already silently dropped at runtime — this release simply makes
that reality explicit at the schema level. No migration script needed.

The `--strict` flag that would reject unknown fields is deferred to v1.10.x as opt-in.

### Upgrading

```bash
go install github.com/had-nu/wardex@v1.9.1
```

No action required for existing pipelines. Optional housekeeping: align your local
`wardex-config.yaml` against the updated example at `doc/examples/wardex-config.yaml`.

### What's next

The deferred questions captured in `internal/SPEC_v1.9.1_schema_cleanup.md` (Open
Questions section) feed the v1.10.x design discussion: re-introduction of
`organization` as audit metadata coupled to report output, opt-in `--strict` field
validation, and whether `domain_weights` can be wired into the scorer with a
reproducible cross-organisational use case.
