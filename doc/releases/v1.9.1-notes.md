## Wardex v1.9.1 — Schema Consistency Cleanup

Patch release that aligns `wardex-config.yaml` with what the code actually consumes.
No behavioural change. No migration required.

The audit that prompted this release found six configuration fields declared in the
schema but never read by any active code path. Users who calibrated those fields
carefully observed no effect — a footgun for any organisation adopting Wardex,
regardless of sector or maturity. v1.9.1 removes the orphan fields, corrects the
documentation that mis-taught them, and adds a CI guard-rail that prevents this
class of regression from recurring.

### Removed

The following fields had no consumer in the active codebase. They have been removed
from `config/config.go` and `pkg/model/release.go`, and from documentation/examples:

- **`organization`** (`name`, `sector`, `scope`) — informational only; never read.
- **`domain_weights`** — placeholder for an unshipped feature; removed pending a
  concrete use case.
- **`control_weights`** — placeholder; same rationale.
- **`thresholds`** (top-level: `fail_above`, `warn_above`) — duplicated
  `release_gate.warn_above` semantically and was never read. The live threshold field
  is `release_gate.warn_above`. The live failure-on-gap flag is `--fail-above` on
  `wardex evaluate` (CLI), not a YAML field.
- **`reporting.verbose`** — the CLI flag `--verbose` in `main.go` is the source of
  truth.
- **`release_gate.asset_context.data_class`** — declared in `model.AssetContext` but
  never consumed by the scorer (which reads `criticality`, `internet_facing`,
  `requires_auth`, `environment`). Re-introduction with proper scoring semantics for
  data classification is deferred to v1.10.x.

### Fixed

- **`doc/examples/wardex-config.yaml`** has been rewritten with fields verified
  against the live structs. The v1.9.0 file taught a `thresholds:` block with keys
  (`critical_vulnerabilities`, `high_vulnerabilities`) that did not match any field
  in the codebase. The new file mirrors `Config`, `model.AssetContext`, and
  `model.CompensatingControl` exactly.
- **`test/testdata/wardex-config.yaml`** had three classes of error: orphan blocks
  (`organization:`, `domain_weights:`), wrong key (`accept:` — the schema field is
  `acceptance:`, and the HMAC secret comes from the `WARDEX_ACCEPT_SECRET` env var
  not from YAML), and `compensating_controls` using `id`/`name`/`reduction` instead
  of the actual fields `type`/`effectiveness`/`justification`. All corrected.

### Added

- **`config/config_examples_test.go`** — `TestPublishedExamplesMatchSchema` runs the
  YAML decoder with `KnownFields(true)` against `doc/examples/wardex-config.yaml`
  and `test/testdata/wardex-config.yaml`. Any field in those files that does not
  match a yaml tag in `Config` (or its nested structs) fails CI. This is the
  guard-rail that prevents a schema-versus-example drift from shipping again.

### Notes

- `internal/SPEC_wardex_trust_.md` contained an example
  `domain_weights.technological: "PENDING_APPROVAL"` for the `wardex config seal`
  command. That reference is now historical; the SPEC has been annotated to point
  readers at a live field (`release_gate.risk_appetite`) for equivalent examples.
- A regression test in `config/config_v191_test.go` ensures legacy YAMLs (with the
  now-removed blocks, including `data_class`) continue to parse without error in
  production code paths.

### Backward compatibility

`wardex-config.yaml` files written for v1.9.0 with any of the removed blocks
continue to load without error in v1.9.1 production code paths. The runtime
`Load()` continues to accept unknown fields (Go YAML decoder default), so existing
pipelines do not break. The strict guard-rail (`KnownFields(true)`) applies only to
files **shipped as published examples** by the project itself.

The opt-in `--strict` flag for end-user CI validation is deferred to v1.10.x.

### Upgrading

```bash
go install github.com/had-nu/wardex@v1.9.1
```

No action required for existing pipelines. Optional housekeeping: align your local
`wardex-config.yaml` against the updated example at `doc/examples/wardex-config.yaml`.

### What's next

The deferred questions captured in `internal/SPEC_v1.9.1_schema_cleanup.md` (Open
Questions section) feed the v1.10.x design discussion: re-introduction of
`organization` as audit metadata coupled to report output, opt-in `--strict` field
validation, data-classification-driven scoring, and consolidation of
`model.AssetContext` and `model.AssetExposureContext` (which today carry overlapping
concepts under different YAML tags — `data_class` vs `data_classification`).
